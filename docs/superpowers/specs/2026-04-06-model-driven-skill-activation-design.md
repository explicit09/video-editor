# Model-Driven Skill Activation — Design Spec

## Problem

The in-app AI agent has 9 detailed workflow skills on disk (2000+ lines of step-by-step guidance) but can barely access them. The current `SkillRegistry` uses brittle keyword matching — "how many episodes" doesn't trigger `podcast-episode-producer` because "episodes" isn't in the keyword list. Meanwhile, external MCP agents (Claude Code) get the same skills and use them effectively because the model decides when to invoke them.

Result: the in-app agent picks wrong tools (`detect_episodes` instead of `analyze_transcript`), uses wrong strategies (`remove_section` 8x instead of `extract_segment`), and lacks the safety boundaries encoded in the skills.

## Solution

Replace keyword-based skill matching with model-driven activation. Add an `activate_skill` tool that the model calls when it recognizes a workflow match. Include a skill catalog (names + descriptions) in the system prompt so the model always knows what's available.

This mirrors how external agents work: the model sees available skills, decides when one applies, and activates it.

---

## The `activate_skill` Tool

**Definition:**

```
name: activate_skill
description: Activate a workflow skill for the current task. Returns the full 
workflow with step-by-step instructions, tool usage rules, and safety boundaries. 
Always activate the relevant skill before starting a complex editing workflow.
params:
  name (string, required): Skill name from the available skills list
```

**Resolver:** Returns `[]` — handled upstream in AIChatController.

**Handler behavior:**
1. Look up skill via `SkillRegistry.skill(named:)`
2. If not found, return error with list of available skill names
3. If found, return the full SKILL.md content + recommended tools note
4. Set `activeSkill` for UI display

**Tool result format:**

```
Skill activated: podcast-episode-producer

Recommended tools for this workflow: analyze_transcript, extract_segment, 
auto_cut, set_overlay_config, ...
(Additional tools beyond this list are available if needed.)

---
[Full SKILL.md content here]
```

The model reads the workflow and follows it. It can call `activate_skill` again with a different skill name to switch workflows mid-conversation.

## System Prompt Changes

### Add skill catalog

A new `<available_skills>` block in `ClaudeProvider.systemPrompt`, built dynamically from `SkillRegistry`:

```xml
<available_skills>
When you recognize a task matches one of these workflows, call activate_skill 
to get the full step-by-step workflow before proceeding.

- podcast-episode-producer: Produce edited podcast episodes from raw recordings...
- auto-cutter: Extract and clean podcast episodes...
- rough-cut-assembler: Assemble raw footage into a rough cut...
- beat-sync-editor: Edit video to music by syncing cuts to beats...
- pacing-optimizer: Remove dead zones, tighten silence, adjust speed...
- shorts-formatter: Format clips for vertical platforms...
- viral-clip-extractor: Find 30-60s viral clips from long-form video...
- podcast-editor: Clean up podcast/interview recordings...
- meeting-highlights: Extract decisions and action items from meetings...
</available_skills>
```

This catalog is generated by `SkillRegistry.skillCatalog()` — not hardcoded. Skills added or removed on disk are reflected automatically.

### Inject catalog into system prompt

`ClaudeProvider` needs access to the skill catalog string. It gets passed in during initialization or via a callback, since `ClaudeProvider` lives in the AIServices package and can't access `SkillRegistry` directly from the app layer.

Approach: `ClaudeProvider` gets a new mutable property `var skillCatalog: String = ""`. `AIChatController` sets `provider.skillCatalog = skillRegistry.skillCatalog()` after loading skills. The system prompt getter reads this property to include the catalog block. If empty, the block is omitted.

---

## Tool Set Management

No tool filtering. All tools are always available. When a skill is activated, the skill content itself guides which tools to use at each step ("Step 1: call `analyze_transcript`"). The `allowed-tools` field in skill frontmatter becomes informational — included in the tool result so the model knows the curated set, but not enforced as a filter.

This means:
- No tool filtering logic
- No `activeSkillTools` state
- The skill content IS the guidance
- The model can always call any tool if the workflow requires something unexpected

## Removing Old Keyword Matching

### Remove from AIChatController

The current keyword-matching block (approximate lines 192-206):

```swift
var skillPrompt: String? = nil
if let skill = skillRegistry.match(message) {
    activeSkill = skill.name
    skillPrompt = skill.content
    if !skill.tools.isEmpty {
        selectedTools = allAvailableTools.filter { skill.tools.contains($0.name) }
    }
    if skill.model == "standard" {
        currentModel = IntentRouter.ModelTier.standard.rawValue
    }
} else {
    activeSkill = nil
}
```

This entire block is removed. Also remove:
- `skillPrompt` variable
- `additionalSystemPrompt: skillPrompt` parameter in the `complete()` call
- The `<skill>` injection in `ClaudeProvider.complete()`

### What stays

- `SkillRegistry` class — still loads skills from disk
- `EditingSkill` struct — unchanged
- Skill files on disk — unchanged, same source of truth for both in-app and external agents
- `activeSkill` property — set by the `activate_skill` handler instead of keyword matching
- `SkillRegistry.match()` and `matchScore()` — can remain for backward compatibility but are no longer called

|------|--------|

| `SkillRegistry.swift` | Add `skillCatalog() -> String` method |

## What Does NOT Change

- Skill files on disk — same SKILL.md files used by both surfaces
- SkillRegistry loading — still parses frontmatter and content
- EditorIntent enum — no changes
- MCPServer — no changes (MCP clients already use skills via Claude Code's native skill system)
- IntentRouter — no changes
- Tool definitions — all existing tools unchanged
